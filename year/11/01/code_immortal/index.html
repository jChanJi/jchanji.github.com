代码不朽：编写可维护软件的10大要则（Java版）

Nov 01, 2018 838

前言

往往很多人注重自己的编程思维的锻炼、业务需求理解能力的提升、编程框架原理、网站架构的理解等等所谓的上层建筑，但是作为一位合格的程序员基本的要求是很够写出让别人容易理解，让自己思路清晰，让bug尽量减少的代码。

在阅读过代码不朽这本书后，对代码整洁的认证就多了一个认知的标准，文中通过具体的数字来对所写的代码进行约束，这样就使我们对自己的代码有了一把衡量的尺度。也许有些标准确实是很难遵循，那也无伤大雅，我们需要做的就是努力的向标准逼近。这些看似简单的标准在实际编码中可能尤为困难。

本文并没有将书中所有的规则都罗列出来，挑选了比较重要且体会比较深的几条。

1. 编写短小的代码单元

  1. 代码单元的长度限制在15行以内
  2. 不写重复的代码

2. 编写简单的代码单元

每个代码的分支点数量不超过4个,即if，else的等条件语句的分支个数不宜超过4个

3. 代码单元的接口简单

  1. 每一行代码的参数个数不能超过4个
  2. 将多个参数提取成对象

4. 分离模块（类）之间的关注点

保持类的体积尽可能的小，并限制外部类的调用数量<br>
  1. 根据不同的关注点拆分类（尽量使类的职能单一）
  2. 隐藏接口背后的特定实现（实现接口，接口中写公共方法）
  3. 使用第三方库，框架替换自定义代码
  4. 一个接口至少被两个类实现，否则不要使用接口

5. 架构组件松耦合

  1. 尽量减少当前模块中需要暴露给其他组件模块的相关代码
  2. 透传代码必须避免（将自己的代理暴露给客户），就如同你向一个公司的帮助中心咨询了一个问题，他没有给出答案而是将问题转给了另外一个公司

6. 保持架构组件之间的平衡

保持代码中组件的数量接近7（6-12）

7. 保持小规模代码库

要控制需求的蔓延，难以理解已有的代码，进而畏惧触碰代码，这是开发人员选择复制粘贴代码的通常理由

8. 编些简洁的代码

  1. 不要编写单元级别的代码坏味道
  2. 不要编写不好的注释
  3. 不要注释代码
  4. 不要保留废弃的代码
  5. 不要使用过长的标识符名称
  6. 不要使用魔术常量
  7. 不要使用未正确处理的异常
